# Cleaver 拆解结果：美食 Tinder（餐厅滑卡选择器）

## 概述

用户描述了一个类似 Tinder 的餐厅发现 App：全屏卡片、左右滑动选择餐厅、大图 + 餐厅名 + 评分 + 距离，滑到喜欢的收藏。这是一个典型的"决策疲劳杀手"型产品——把选择餐厅从列表筛选变成直觉式的快速判断。

由于用户只提供了口头描述（无具体产品名称），我基于描述中的核心特征进行拆解，选择 **Fast Track** 路径（1 个 Foundation Prompt + 2-3 个功能 Prompt），约 30 分钟可出原型。

---

## Phase 1: 理解产品

### 产品核心
- **一句话**: 用 Tinder 式滑卡交互做餐厅发现，让"吃什么"变成一个快速直觉判断而非理性筛选
- **目标用户**: 选择困难症患者、想吃新餐厅但懒得翻大众点评的人
- **核心问题**: 传统美食 App 信息过载，用户只需要一个快速的是/否判断

### 关键交互特征（从描述中提取）
1. **全屏卡片** — 一次只看一个选择，强迫专注
2. **左右滑动** — 身体语言式决策（左=不要，右=要）
3. **大图主导** — 视觉优先，先看菜再说
4. **信息精简** — 只留餐厅名、评分、距离三个关键信息
5. **收藏机制** — 右滑=收藏，简单清晰的输出

### 设计语言推断
- 移动优先（滑动手势天然是移动端）
- 沉浸式（全屏卡片 = 无干扰）
- 视觉驱动（大图 > 文字）
- 轻松有趣（不是严肃的美食评测工具）

---

## Phase 2: 推荐路径

**选择 Fast Track**（1 Foundation + 2 功能 Prompt），理由：
- 产品核心交互单一（滑卡），不需要逐组件拆解
- 用户描述已经很聚焦，不是大型项目
- 快速出原型最能验证这个想法

如需更深入拆解（Standard Build 5-8 prompts），可以随时切换。

---

## The Prompts

### Prompt 0: Foundation（基础设施层）

> 帮我搭一个 React Native + Expo 项目，做一个美食发现 App。整个 App 的核心就是全屏卡片滑动——像 Tinder 那样，但选的是餐厅。
>
> 技术栈：Expo managed workflow，TypeScript，用 react-native-deck-swiper 处理滑卡手势。样式用 StyleSheet（不要引入额外 UI 库），深色背景 + 暖色调点缀。
>
> 项目结构：
> ```
> /app
>   /components
>     RestaurantCard.tsx   -- 全屏卡片组件
>     SwipeIndicator.tsx   -- 左右滑动提示
>     MatchAnimation.tsx   -- 匹配/收藏动画
>   /screens
>     HomeScreen.tsx       -- 主滑卡页面
>     CollectionScreen.tsx -- 收藏的餐厅列表
>   /data
>     restaurants.ts       -- mock 餐厅数据
>   /types
>     index.ts             -- 类型定义
> ```
>
> 数据模型：Restaurant { id, name, image, rating, distance, cuisine, priceRange }
>
> 不要做：导航、登录、搜索、设置页面。这个阶段只做滑卡这一个核心交互。
>
> 你会知道完成了，当：App 打开就是全屏餐厅卡片，可以左右滑动，右滑有收藏反馈。

**What this adds:** 项目骨架、技术选型、目录结构、数据模型——所有后续 prompt 的基础。
**Why this prompt works:** 先把"是什么"和"不做什么"说清楚，避免 AI 过度设计。"不要做登录/搜索"这个 not-to-do 是关键，否则 80% 概率会塞一堆不需要的东西。
**Pro tip:** Foundation prompt 的核心价值是建立约束。越是简单项目，越需要明确边界——因为简单项目最容易膨胀。

---

### Prompt 1: 核心滑卡体验

> 在 Foundation 的基础上，做主页面——全屏餐厅滑卡。
>
> 卡片占满整个屏幕（除了顶部一个小的状态栏区域）。每张卡片：
> - **背景是一张大图**，铺满卡片，底部有从黑到透明的渐变遮罩，让文字在图上清晰可读
> - **底部叠加信息**：餐厅名（大号粗体白色）、评分（黄色星星 + 数字）、距离（"800m" 这种简洁格式）、菜系标签
> - **滑动提示**：手指按住时卡片微微倾斜跟随手指方向，松手时如果滑动距离不够就弹回
> - **右滑收藏反馈**：卡片飞出屏幕右侧时，右上角出现一个绿色的 "SAVED" 标签，短暂显示后消失
> - **左滑跳过**：卡片飞出左侧，无额外反馈，下一张卡片从底部弹出
> - **卡片堆叠感**：后面能看到下一张卡片的边缘（露出 10-20px），形成"还有更多"的期待感
>
> 交互感觉要参考 Tinder：流畅、跟手、有弹性。不要做成按钮点击式的——必须是手指拖拽滑动。
>
> 用 mock 数据（10-15 家餐厅，用 unsplash 的美食图片 URL）。
>
> 不要做：真实的后端 API、用户认证、分享功能。

**What this adds:** 产品的灵魂——滑卡交互。这是用户第一眼看到、第一次触摸的东西。
**Why this prompt works:** 分三层描述——视觉（大图+渐变+文字叠加）、交互（跟手拖拽+弹性回弹）、反馈（收藏动画）。每层都用"用户感受"而非"技术实现"来描述。"参考 Tinder"这一句话比 10 行交互动效描述都管用。
**Pro tip:** "跟手"和"有弹性"这两个词比任何具体的动画参数都有效——AI 理解这种感觉并会自动选择合适的 spring 动画参数。

---

### Prompt 2: 收藏列表 + 收尾打磨

> 现在加一个收藏列表页面，和一些打磨细节。
>
> **收藏列表**（底部导航第二个 tab，图标用 ♥）：
> - 简洁的竖向列表，每行是：缩略图（左边小方块）+ 餐厅名 + 评分 + 距离
> - 可以左滑删除（跟主页面滑卡的逻辑呼应）
> - 顶部显示 "已收藏 X 家" 计数
> - 空状态：友好的提示 "还没有收藏？去滑一滑吧" + 一个跳转按钮回主页
>
> **打磨细节**：
> - 主页顶部显示剩余卡片数量 "还有 5 家待发现"（给人紧迫感）
> - 滑完所有卡片时，显示一个 "今日发现完毕" 的结束页面，带一个 "再看一轮" 按钮
> - 卡片滑出时的声音反馈（可选，用 expo-av 播放一个简短的 "swoosh" 音效）
>
> 底部导航两个 tab：主页面（🏠）和收藏（♥），简洁的顶部栏。
>
> 不要做：餐厅详情页、地图、评论区、分享到社交媒体。

**What this adds:** 完整的用户闭环——从发现到收藏到回顾。加上空状态和结束状态的处理，让 App 感觉完整而非半成品。
**Why this prompt works:** 在核心交互完成后才加次要功能，保证每一步都是可工作的。空状态和结束状态是容易被忽略但直接影响"完成感"的细节。"左滑删除"与主交互呼应，形成一致的交互语言。
**Pro tip:** 永远在最后一个 prompt 处理空状态和边界情况。先做核心再补边角，顺序很重要——如果一开始就处理空状态，AI 会花一半精力在不重要的分支上。

---

## 建议的使用顺序

1. **Prompt 0** → 搭好项目骨架，确认能跑起来
2. **Prompt 1** → 实现核心滑卡，在手机上试一试手感
3. **Prompt 2** → 加收藏列表和打磨

如果想更快，可以把 Prompt 0 和 1 合并成一个 prompt——虽然会稍微不可控，但能更快看到效果。

## 如果想进一步迭代

- **加个性化推荐**: 根据右滑历史推荐相似菜系
- **加地图视角**: 收藏的餐厅在地图上标注
- **加社交元素**: 看朋友收藏了什么
- **加商家端**: 餐厅可以付费让自己的卡片更早出现

---

*拆解完成。三个 prompt，约 30 分钟出可交互原型。核心洞察：这个产品的灵魂不是餐厅信息，而是"让选择变成直觉"的交互体验——所以 Prompt 1（滑卡）是最关键的。*
